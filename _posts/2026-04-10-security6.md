# 📖 오늘 배운 내용

> **무차별 대입 공격의 원리를 깊이 이해하고자, 먼저 의도적으로 취약한 SSH 환경을 구성하여 그 위험성을 직접 분석해 보았습니다. 이후 실질적인 방어를 위해 비밀번호 대신 공개키 인증 방식으로 전환하였으며, 여기에 Fail2Ban을 도입하여 반복적인 비정상 로그인 시도를 네트워크 단에서 자동으로 차단하는 견고한 2중 보안 아키텍처를 실습하였습니다.**

## 💡 핵심 개념 및 실습 흐름
* **1단계: 비밀번호 인증의 취약점 분석**
  아이디와 비밀번호 기반의 기본 SSH 설정이 공격자의 자동화된 스크립트(Bot)와 무차별 대입 공격에 얼마나 취약한지 직접 테스트하며 보안 위협을 확인했습니다.
* **2단계: 공개키(비대칭키) 인증 기반 접근 통제**
  취약점을 근본적으로 보완하기 위해 SSH 설정 파일(`sshd_config`)을 수정하여 비밀번호 입력을 원천 차단했습니다. 서버의 공개키와 관리자의 개인키가 일치할 때만 접속을 허용하는 시스템으로 전환하여 불법적인 접근 시도를 방어하였습니다.
* **3단계: Fail2Ban을 활용한 능동적 방어**
  키 인증 방식을 적용한 후에도 끈질기게 발생하는 무의미한 로그인 시도와 리소스 낭비를 막기 위해 Fail2Ban을 구축했습니다. 인증 실패 로그를 실시간으로 모니터링하고, 설정한 횟수를 초과하는 IP를 리눅스 방화벽(`iptables`)에 자동 등록하여 영구 차단하는 능동적인 방어 로직을 구현하였습니다.

## 🛠️ 실전 적용 핵심 명령어

```bash
# 1. 키 기반 인증 체계 구축
ssh-keygen                          # 비대칭키(공개키/개인키) 쌍 생성

# 2. SSH 서버 설정 강화 (취약점 제거)
sudo nano /etc/ssh/sshd_config      # SSH 설정 파일 편집
# [설정 변경] PasswordAuthentication yes -> no (비밀번호 접속 차단)
# [상태 확인] PubkeyAuthentication yes (공개키 접속 허용)
sudo systemctl restart ssh          # 변경된 SSH 보안 설정 적용

# 3. Fail2Ban 모니터링 및 방어 제어
sudo systemctl restart fail2ban     # Fail2Ban 서비스 재시작
sudo fail2ban-client status sshd    # 현재 감시 상태 및 차단된 악성 IP 목록 확인

# 4. 방화벽 룰 적용 확인
sudo iptables -nL                   # 방화벽 단에서 해당 IP가 정상적으로 DROP/REJECT 되었는지 확인
```
